Manipuladores de Retorno de Chamada

Essa sessão descreve como o padrão ReactJS' <div onclick={this.handleClick} /> é transposto em ReasonReact.

Callback Sem Atualização de Estado #

Dois cenários.

Sem Leitura Do self #

Lembrete: self é o this do ReasonReact. É um record que contém coisas como state, retainedProps entre outros.

Se você estiver apenas passando adiante uma callback prop para o seu nodo filho, você faria exatamente a mesma coisa que você faria em ReactJS:

let component = ...;

let make ::name ::onClick _children => {
  ...component,
  render: fun self => <button onClick=onClick />
};

Nenhuma surpresa aqui. Tendo o JSX do Reason <a href=[sintaxe punning](https://reasonml.github.io/guide/language/jsx#punning), o `button` irá ser formatado para `<button onClick />`.

Similarmente, para pre-processar um valor antes de enviá-lo de volta para o componente pai:

let component = ...;

let make ::name ::onClick _children => {
  let click event => onClick name; /* passa a string name para o component pai */
  {
    ...component,
    render: fun self => <button onClick=click />
  }
};

Lendo dentro de self #

Para acessar state, retainedProps e outros itens dentro de self em uma callback, você precisa encapsular o callback em uma camada extra chamada self.handle:

let component = ...;
let make ::name ::onClick _children =>
  let click event self => {
    onClick event;
    Js.log self.state;
  };
  {
    ...component,
    initialState: ...,
    render: fun self => <button onClick=(self.handle click) />
  }
};

Nota como seu callback click agora leva um argumento extra self. Formalmente, self.handle espera um callback que

  • aceite apenas um payload que você normalmente passaria diretamente para por exemplo handleClick,
  • plus the argument self,
  • retorne "nada" (vulgo (), vulgo unit).

Nota 2: algumas vezes você pode estar encaminhando handle para algumas funções auxiliares. Passar todo o self em vez de anotá-lo. Isso evita um comportamento de tipo de record self </ code> complexo. Veja <a href="record-field-reduce-handle-not-found.md">Campo De <i>Record</i> <code>reduce/handle Não Encontrado</a>.

Explicação #

Na realidade, self.handle é apenas uma função regular aceitando dois argumentos, o primeiro sendo o callback em questão, e o segundo sendo o payload que se deseja passar no callback.

Sacou? Através do currying à nível-de-linguagem natural, nós usualmente só iremos pedir para você passar o primeiro argumento. Isso retorna uma nova função que leva o segundo argumento e executa no corpo da função. O segundo argumento sendo passado por quem o está chamando, também conhecido como o componente que você está renderizando!

Callback Recebendo Múltiplos Argumentos #

Algumas vezes, o componente que você está chamando vem do Javascript (usando ReasonReact<->ReactJS interop), e é a callback prop que pede a você para passar a callback que recebe mais de um argumento. Em ReactJs, pareceria como:

handleSubmit: function(username, password, event) {
  this.setState(...)
}
...
<MyForm onUserClickedSubmit={this.handleSubmit} />

Você não pode escrever tal handleSubmit em ReasonReact, como handle espera que seja encapsulado em torno de uma função que só leve um argumento. Aqui está uma solução:

let handleSubmitEscapeHatch username password event => self.handle (fun tupleOfThreeItems => doSomething tupleOfThreeItems) (username, password, event);
...
<MyForm onUserClickedSubmit=(handleSubmitEscapeHatch) />

Basicamente, você escreveria um callback normal que:

  • leve todos esses argumentos do prop do callback do componente JS
  • embale-os dentro de uma tupla e chame self.handle,
  • passe para handle a função usual que espera um argumento simples,
  • termine chamando self.handle passando a tupla você mesmo.

Callback Com Atualização de Estado #

Você não pode atualizar o estado em self.handle; ao invés disso você precisa usar self.reduce.
